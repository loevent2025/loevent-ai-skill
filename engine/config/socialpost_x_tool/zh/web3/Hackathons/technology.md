# HACKATHONS × TECHNICAL / 黑客松 × 技术类

## 适用场景
开发者主导的黑客松、协议黑客松、基础设施挑战、安全黑客松、ZK/L2 专项活动、智能合约竞赛、技术创新挑战

## 场景特征
- **规模**：50-300 人
- **时长**：24-72 小时
- **形式**：极限开发、技术评审、代码审查、架构讨论
- **核心价值**：技术深度、代码质量、创新方案、技术人脉

## 受众定位
- **主要受众:** 核心开发者 / Core Developers - 有经验的工程师，追求技术挑战
- **次要受众:** 协议贡献者 / Protocol Contributors - 开源贡献者，想深入特定技术栈
- **第三受众:** 技术创业者 / Technical Founders - 验证技术方案可行性
- **扩展受众:** 学习者 / Learners - 想通过实战学习的开发者
- **受众行为:** 挑战技术难题、学习新技术栈、展示技术能力、寻找技术伙伴

### 常用受众组合
- **Expert Track:** 核心开发者为主，适合高难度技术挑战
- **Learning Track:** 学习者 + 核心开发者，适合有导师指导的 Hackathon

## 内容元素
- 技术主题与挑战定义
- 技术赛道设置
- 评审标准（技术权重）
- 导师阵容（技术背景详述）
- 技术资源（SDK、API、文档、测试网）
- 代码提交要求

## 语言处理
- 技术问题导向开场
- 强调代码质量而非仅是 Demo
- 技术术语精准使用
- 挑战难度适度说明
- 学习资源同步提供

## 结构策略
- 开场信息块占比 15%
- 技术挑战块占比 35%：核心问题 + 赛道 + 评审标准
- 资源支持块占比 25%：导师 + 技术资源
- 参与说明块占比 15%
- 行动号召块占比 10%
- Emoji 密度控制：总量 8-10 个，偏简洁

## 情绪氛围
技术严谨、挑战驱动、以开发者为中心。“用代码解决难题”的氛围。

## 风格变体

### 变体A：协议挑战型
- **适用：** 特定协议技术挑战
- **开场格式：** 🔧 {协议名} Technical Challenge | 解决这些问题
- **语气：** 聚焦、深入、技术导向

### 变体B：技术专项型
- **适用：** 单一技术方向（ZK、安全、性能）
- **开场格式：** ⚡ {技术主题} Hackathon | 用代码攻克难题
- **语气：** 专业、挑战、硬核

## 对比句式库
- 不比谁的 PPT 漂亮，只看代码能不能跑
- 评审看的是架构和实现，不是故事
- 技术深度 > Demo 效果
- 解决真问题，不造假需求

## 注意事项

### 技术类 Hackathon 要点
1. **技术挑战要具体**：不能泛泛说"创新"，要有具体问题
2. **评审标准技术权重高**：代码质量 > Demo 效果
3. **技术资源要充足**：SDK、文档、测试环境、导师
4. **难度要说明**：让参与者评估自己是否适合

# WEB3 DEMO DAY × TECHNICAL

## Applicable Scenarios
Web3 technical project showcases, open-source protocol demo days, on-chain tool exhibitions, hackathon technical track result showcases, smart contract security tool demos, developer tool launches

## Scenario Characteristics
- **Format:** Technical demos + live on-chain interaction demos + technical Q&A + developer exchange
- **Core Value:** Technical showcase, solution evaluation, developer adoption, technical credibility

## Audience Profile
- **Primary Audience:** Web3 developers — discovering new tools, learning technical approaches, evaluating integrations
- **Secondary Audience:** Protocol architects and tech leads — assessing technology, finding solutions to real engineering problems
- **Tertiary Audience:** Technical investors — evaluating technical feasibility and innovation
- **Extended Audience:** Open-source community — discovering projects, contributing, starring and forking
- **Audience Behavior:** Evaluating technology, learning approaches, finding tools, exchanging technical insights, finding hackathon inspiration

### Common Audience Combinations
- **Developer Focus:** Developers + open-source community — ideal for tool showcase events
- **Evaluation Focus:** Tech leads + investors — ideal for in-depth technical showcases

## Content Elements
- Technical demo day theme and positioning
- Presenting project technical directions (Infra, DeFi protocols, security tools, SDKs, etc.)
- Technical reviewer lineup (core protocol developers + prominent builders)
- Live demo format (contract interaction, architecture walkthroughs, performance benchmarks)
- Technical Q&A session
- Open-source resources (GitHub, documentation, SDK)

## Language Guidelines
- Technically grounded, but direct and accessible
- Emphasize "contracts that actually run," "live on-chain demo," and "show me the code"
- Technical details are the differentiator (gas optimization, TPS, security)
- Developer community atmosphere throughout
- Practical over impressive — focus on what problems get solved, not how clever the code is

## Content Structure
- Opening info block: 15%
- Technical projects block: 35% — project direction + technical highlights
- Reviewer lineup block: 20% — technical backgrounds + representative work
- Value proposition block: 20% — what you'll learn + what you can use
- Call to action block: 10%
- Emoji density: 8–12 total, clean and technical

## Tone & Atmosphere
Technically rigorous, demo-driven. "Show me the code, show me on-chain" builder energy.

## Attendee Tips

### For Developers
- Come with your own technical problems and development pain points in mind
- Focus on tools and solutions most relevant to your current project
- Go deep on technical implementation details after each demo
- Connect with project maintainers on Telegram or Discord for ongoing technical exchange
- Try out interesting tools and give the team direct feedback

### For Tech Leads
- Focus on projects that solve real engineering problems
- Evaluate technical maturity, audit status, and maintenance activity
- Understand the differences from your current stack and the migration cost
- Consider integration feasibility and pilot testing opportunities
- Check GitHub star count, contributor count, and commit frequency as signals

### After the Event
- Star and fork projects that interest you
- Set them up and test them in your local environment
- Join the project's Discord technical channels
- Submit issues and PRs — contribute to the open-source work
- Share your evaluation and impressions with your technical community

## Notes

### Key Principles for Technical Demo Days
1. **Live Demo Is Non-Negotiable:** Contracts must run in real time; on-chain interaction must be demonstrated live
2. **Technical Depth Is Expected:** Architecture, gas consumption, TPS, and security audit status all matter
3. **Reviewers Must Understand the Tech:** They need to ask the right questions about contract security and performance bottlenecks
4. **Developer Perspective First:** Always answer — what development pain point does this solve? How is the developer experience?
5. **Resources Must Be Provided:** GitHub, documentation, SDK, and Discord technical channels
6. **Open Source by Default:** Projects should ideally be open-source so the community can evaluate and contribute

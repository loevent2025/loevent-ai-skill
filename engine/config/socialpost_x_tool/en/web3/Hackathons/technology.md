# HACKATHONS × TECHNICAL

## Use Cases
Developer-led hackathons, protocol hackathons, infrastructure challenges, security hackathons, ZK / L2 specialized events, smart contract competitions, technical innovation challenges

## Scene Characteristics
- **Scale:** 50–300 people
- **Duration:** 24–72 hours
- **Format:** Intensive development, technical judging, code review, architecture discussion
- **Core Value:** Technical depth, code quality, innovative solutions, technical networking

## Audience Targeting
- **Primary Audience:** Core Developers — experienced engineers seeking technical challenges
- **Secondary Audience:** Protocol Contributors — open-source contributors looking to go deep on a specific tech stack
- **Tertiary Audience:** Technical Founders — validating the feasibility of a technical approach
- **Extended Audience:** Learners — developers who want to learn through hands-on practice
- **Audience Behavior:** Tackle hard technical problems, learn new tech stacks, demonstrate technical ability, find technical collaborators

### Common Audience Configurations
- **Expert Track:** Primarily core developers — best for high-difficulty technical challenges
- **Learning Track:** Learners paired with core developers — best for mentor-guided hackathons

## Content Elements
- Technical theme and challenge definition
- Technical track structure
- Judging criteria (with technical weighting)
- Mentor lineup (with detailed technical backgrounds)
- Technical resources (SDK, API, documentation, testnet)
- Code submission requirements

## Language Tone
- Open with a technical problem statement
- Emphasize code quality, not just a working demo
- Precise use of technical terminology
- Clearly communicate challenge difficulty level
- Provide learning resources in parallel

## Content Structure
- Opening info block: 15%
- Technical challenge block: 35% — core problems + tracks + judging criteria
- Resource support block: 25% — mentors + technical resources
- Participation details block: 15%
- Call-to-action block: 10%
- Emoji density: 8–10 total, lean and clean

## Emotional Tone
Technically rigorous, challenge-driven, developer-centered. "Solve hard problems with code" energy.

## Style Variants

- **Best for:** Technical challenges around a specific protocol

## Notes

### Key Principles for Technical Hackathons
1. **Technical challenges must be specific:** Don't say "innovate" — define concrete problems to solve
2. **Judging criteria should heavily weight technical merit:** Code quality > demo polish
3. **Technical resources must be sufficient:** SDK, documentation, test environments, mentors
4. **Communicate difficulty level clearly:** Let participants assess whether it's right for them

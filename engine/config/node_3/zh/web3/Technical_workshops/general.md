# 筛选维度

---

## 维度1：网络与技术基础设施（第一硬性条件）

**定义：**
Web3 Technical Workshop的网络风险与AI Commercial Workshop不同：带宽通常不是瓶颈，核心威胁是境外区块链节点被防火墙屏蔽。学员在实操阶段需要同时访问测试网RPC节点、部署合约、与链上合约交互——一旦节点不可访问，全体学员的实操进度同时中断，且无法通过"换个练习"来绕过。

**考察方向：**

网络带宽与并发
- 总带宽 ≥200Mbps（50人以上建议500Mbps）
- 支持全员（100%在线率）同时访问链上节点和RPC服务
- 有线+无线双覆盖（实操区优先有线）
- 有线接入点：每2–4人1个有线接口
- WiFi无死角覆盖，AP数量与学员密度匹配

访问权限（Web3核心要求）
- 无端口限制、无防火墙屏蔽
- 可正常访问主流测试网RPC节点：Ethereum Sepolia/Goerli、Solana Devnet、BNB Testnet、Polygon Mumbai等
- 可正常访问节点服务商：Infura、Alchemy、QuickNode、Ankr
- 可正常访问区块链浏览器：Etherscan、Solscan等（合约验证和交易查询）
- 可正常访问开发依赖：GitHub、npm、crates.io（Rust/Solana开发）、Foundry/Hardhat依赖源
- 允许自带路由器或独立热点接入（强烈建议作为标配备用）

备用方案
- 配备4G/5G热点作为应急备用
- 场地是否允许提前部署本地节点（Ganache/Anvil/Hardhat Network本地链，作为RPC节点不可访问时的降级方案）

电源（与网络并列的硬性条件）
- 每2人至少1个插座（学员通常携带笔记本+手机）
- 每桌配置排插或桌面嵌入式插座
- 讲师区独立供电回路
- 总电路负荷可支持全员设备同时运行


## 维度2：教学空间质量与实操适配性（硬性条件）

**定义：**
Web3 Workshop通常是4–8小时的长时间高密度实操，合约开发、协议调试、链上交互的专注度要求极高。空间质量直接影响学员的状态和学习效果——但Web3社区对空间"够不够精致"的在意程度远低于"够不够用"。

**考察方向：**

桌面与座位
- 桌面宽度 ≥80cm/人（需同时放置笔记本、手账、饮料）
- 桌面深度 ≥60cm（外接键盘/鼠标的使用空间）
- 椅子支持长时间使用（有靠背，非折叠椅）
- 相邻学员肩距 ≥60cm（操作时不相互干扰）

视线与讲师区
- 所有座位可无遮挡看到主屏幕
- 最后排到主屏距离不超过屏幕宽度的6倍（合约代码清晰可读）
- 讲师区有充足走动空间（合约审查和调试辅导时需要走到学员旁边）
- 讲师可方便到达任意学员位置

环境质量
- 空调独立可控（长时间满员使用，温控稳定性要求高）
- 灯光可调节（终端和代码编辑器在高亮环境下有屏幕反光问题）
- 无强反光光源直射屏幕
- 相对安静，与外部噪音有效隔离

## 维度3：讲师演示与教学设备支持能力（硬性条件）

**定义：**
Web3 Workshop的讲师演示内容极度技术密集：Solidity/Rust合约代码、终端命令、链上浏览器实时数据、钱包交互界面——这些内容对屏幕分辨率的要求与AI Commercial Workshop相同，但内容类型决定了白板的使用频率更高（架构图、状态变量、调用栈解释）。

**考察方向：**

屏幕与投影
- 主屏幕/投影分辨率 ≥1080P（合约代码、终端输出必须清晰可读）
- 屏幕尺寸与最远座位距离匹配（建议字体最小12pt在最远处可辨识）
- 支持双屏输出（讲师预览屏 + 学员主屏）
- 支持多信号源切换（讲师笔记本 + 链上浏览器辅助屏）
- HDMI / Type-C 接口齐全，有转接头备用

音响与麦克风
- 音响覆盖全场（超过30人时麦克风是必需项）
- 无线麦克风1支以上（讲师巡回指导时需要）
- 无啸叫风险

白板与辅助工具
- 有白板或可书写展示面（Web3 Workshop中架构图和调用流程图使用频率极高）
- 白板尺寸与空间匹配，所有学员可看清
- 白板笔和擦拭工具供应充足

录制条件（如有录制需求）
- 场地允许全程录制（Web3社区高度依赖课程录像的二次传播）
- 有适合架设摄像设备的位置（主屏+讲师画面）

## 维度4：Web3行业调性匹配（高权重条件）

**定义：**
Web3 Workshop的受众是开发者社区的核心成员——合约工程师、协议开发者、全栈Web3开发者。这个群体对空间调性的判断标准是：够不够"自己人的地方"、够不够极客、够不够开源精神——对过度正式感、培训班感、企业感有明显排斥。这是与AI Commercial Workshop最核心的调性差异。

**Web3 Workshop调性定位：**
> 有极客感，让开发者觉得这是"同类聚集"
> 有开放感，让横向的技术交流自然发生
> 够用就好，不需要为不影响学习效果的装修加分

**✅ 符合的空间类型：**
- 联合办公空间的独立培训区（开放感强，无企业正式感）
- 区块链/Web3公司自有办公区（生态认同感强，开发者有归属感）
- 创意园区改造空间或Loft（工业风、自由氛围契合hack文化）
- 大学计算机系或工程学院教室（开源文化认同度高，去商业感）
- 高校或社区黑客空间（Hackerspace）

**❌ 避免的空间类型：**
- 传统酒店会议室（商务感和消费感过重，与Web3开发者文化强烈冲突）
- 标准化企业培训中心（"打工人培训感"破坏社区横向平等的文化）
- 过度装修的商业场地（Web3社区对形式主义和奢华感有本能排斥）
- 纯消费型场地（商场活动区、网红咖啡馆）


## 维度5：容量与舒适密度控制（硬性条件）

**定义：**
Web3 Workshop是长时间高密度实操，人均空间需求与AI Commercial Workshop相同。Web3开发者对"空间够不够用"的容忍度低，但对"够不够漂亮"的容忍度高——功能优先。

**面积参考标准：**

| 布局类型  | 每人面积    |
| ----- | ------- |
| 课桌式排座 | 1.5–2㎡  |
| 分组岛式  | 2–2.5㎡  |
| U型布局  | 2.5–3㎡  |

**考察方向：**
- 以舒适容量为准，而非场地标注最大容纳人数
- 空调独立可控（长时间满员，温控稳定性要求高）
- 灯光可调节（终端和代码编辑器在高亮环境下有反光问题）
- 卫生间数量（建议50人/1间）
- 茶歇区独立，不与实操区混用

---

## 维度6：时间灵活性与使用弹性（硬性条件）

**定义：**
Web3 Workshop的时间不确定性比AI Commercial更高：合约调试超时、链上交互等待确认、学员提问深度不可控——技术讨论在Web3社区有天然延伸的倾向。

**考察方向：**
- 支持全天使用（9:00–18:00）或半天使用（弹性段）
- 允许延时使用（计费规则是否合理）
- 允许提前1–2小时进场布置和网络调试（RPC节点测试需要时间）
- 活动结束后允许延续自由交流（Web3社区的非正式讨论价值极高，不应被立即驱场）
- 明确最晚撤场时间

多日工作坊场景：需确认同一场地的连续预订可行性，以及每日进场布置的时间保障。

---

## 维度7：交通与到达便利性（高权重条件）

**定义：**
Web3开发者对"为好的活动多跑一段路"的接受度高于一般受众，但工作日全天Workshop的交通便利性仍然影响准时率，迟到会打断讲师节奏和学员实操连贯性。

**考察方向：**
- 距地铁站步行15分钟内（Web3可适当放宽至15分钟）
- 打车可接送（有明确上下车点）
- 夜间离场安全（Workshop结束后的延伸社交常在晚间）
- 场地入口标识清晰（非标准场地需提前规划导航说明）
- 允许放置活动引导标识

## 维度8：社区沉淀与长期复用能力（可权衡条件）

**定义：**
Web3 Workshop对品牌呈现的需求远低于AI Commercial，更看重"社区自己的学习据点"的归属感。如果是系列课程，固定场地能够积累社区记忆，这比品牌视觉呈现更有价值。

**考察方向：**
- 是否允许长期或定期复用（系列课程建立固定社区学习据点）
- 场地方是否本身是Web3生态参与者（生态认同感加分）
- 允许活动全程录制和发布（Web3社区依赖课程录像传播）
- 支持社区留存动作（加群引导、社区扫码）
- 有适合非正式聚集的区域（课间和课后的技术讨论同样有价值）
- 允许社区自带装饰或改造（增强归属感）

## 维度9：运营合规与使用限制（可权衡条件）

**定义：**
Web3 Workshop使用非标准场地的比例高于AI Commercial，使用限制需逐项确认。此外，部分场地对区块链和加密货币相关培训活动有额外限制，需提前核查。

**考察方向：**
- 最大承载人数是否明确并满足需求
- 是否允许举办收费培训活动（部分写字楼或园区有限制）
- 是否对区块链/加密货币主题活动有特殊限制（部分政府/国企场馆存在）
- 是否允许外部餐饮进场（全天Workshop的午餐安排）
- 是否存在噪音限制（实操讨论和调试环节有一定噪音）
- 场地责任险归属是否明确

# 维度默认优先级

## 硬性条件（不满足直接排除）
1. 网络与技术基础设施（尤其：境外RPC节点访问权限、测试网连通性）
2. 教学空间质量与实操适配性（桌面空间、视线、环境）
3. 讲师演示与教学设备支持能力（屏幕清晰度、白板）
4. 容量与舒适密度控制
5. 时间灵活性与使用弹性

## 高权重条件（强烈影响决策）
6. Web3行业调性匹配
7. 交通与到达便利性

## 可权衡条件（结合具体情况判断）
8. 社区沉淀与长期复用能力
9. 运营合规与使用限制

# 混合场景处理指引

### Workshop + Meetup（上午Workshop + 下午Meetup）
- 空间需支持布局快速切换（课桌式 → 开放流动式，翻场时间 ≤1小时）
- 网络带宽按两种场景取高值，RPC访问权限全程保持
- 讲师演示设备与Meetup演讲设备共用，需提前确认兼容性
- 时间弹性需同时满足两种场景，下午Meetup开始时间需留缓冲

### Workshop + Hackathon（培训日 + 开发日）
- 网络带宽标准提升至Hackathon级别（≥500Mbps）
- RPC节点访问权限验证范围扩展至正式网络（Hackathon阶段可能需要主网演示）
- 电源密度升级（Hackathon阶段每工位独立插座）
- 时间弹性升为第一硬性条件（Hackathon开发阶段可能延伸至深夜或通宵）
- 面积按Hackathon标准计算（每人≥2㎡）

### 多轨道并行Workshop（同期多个培训教室）
- 每个教室独立网络回路（避免RPC节点访问带宽相互竞争）
- 每个教室独立空调控制
- 各教室之间需有声学隔离（合约调试和答疑有一定噪音）
- 公共茶歇区面积按总人数规划（多轨道Workshop的茶歇区也是跨轨道社区交流的核心节点）

### 精品小班Workshop（10–25人）
- 布局优先选择U型或岛式（契合Web3横向协作文化）
- 可考虑Web3公司会议室或联合办公独立房间（规模小，调性匹配比设施完善更重要）
- 讲师演示设备要求不变（屏幕清晰度是核心）
- RPC节点访问权限验证同等重要，不因规模小而降低要求

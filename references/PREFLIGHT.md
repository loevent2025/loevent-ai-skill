# 参数 Preflight 分级(所有 skill 共用)

各 skill 的 SKILL.md 用一行指到本文件,定义 Claude 在跑 skill 前**怎么处理参数**:
哪些必须先问、哪些摊个草稿让用户瞄一眼、哪些直接默认别打扰。目标:**既不瞎填、也不过度打断。**

## 核心:跑之前先 preflight 一次,别逐个追问

拿到需求、真正跑脚本**前**,你(Claude)先把这个 skill 的参数过一遍,**把"必问 + 必确认"合并进尽量少的 AskUserQuestion**(能一次问完就别分多次)。已经在 `event.json`/`host.json`/`plan.json` 里有的值直接用,别重复问。

## 三级分级

### 1. 必问(无安全默认,必须先问到才跑)
没有安全默认,且**猜错会污染一个硬事实或对外产出**。**必须问到才跑,绝不静默填默认。**
- 例:活动开始时间(海报会印)、策略 vibe(eventplanner 三选一)、海报风格/平台、主办方官网 URL。

### 2. 必确认(建议默认·可改)
能从 `event/host/plan` **推一个草稿默认值**,但它**会进对外产出**,或**这个 skill 重跑很贵**(出图、已发文案、几分钟的联网调研)。
做法:把推出来的默认值**摊给用户瞄一眼**——一句"我按 X 这么填,要改就说,不改直接走";**让他要改才改,别硬逼他填**。用户说"你看着办"就走默认。
- 例:`event_goal` / `objective` / `GTMmatrix`、竞品数(`max_competitors`)、海报主/副标题。

### 3. 沉默(直接默认,别问)
纯技术细节,有安全默认,**不出现在产出里**;或**这个 skill 根本不消费它**(只留档)。直接用默认/略过,别打断。
- 例:workdir、语言(取 `event.language`)、分辨率/比例的合理缺省、仅留档不参与生成的字段。

## 怎么给一个参数定级(拿不准时按这 4 问)
- **Q1 它会印进/写进给用户看的产出吗?**(海报上的字、文案、方案正文)→ 会 → 必问 或 必确认
- **Q2 缺失时系统会填一个"看着对其实错"的默认吗?**(`00:00`、瞎猜行业)→ 会 → 必问
- **Q3 产出会被刻死、二次重跑很贵吗?**(出图、已发的文案、几分钟调研)→ 会 → 必确认
- **Q4 它是用户的偏好/业务判断,还是纯机械细节 / 本工具压根不消费?** → 判断 → 必问/必确认;纯技术或不消费 → 沉默

**注意:同一个字段在不同 skill 可能不同级。** 例:活动开始时间对 `init` 可空(沉默),对 `poster` 是必问;`event_goal` 在 `audience`/`eventplanner` 里被消费(必确认),在 `company` 里只留档(沉默)。每个 SKILL.md 列出本工具自己的分级。
